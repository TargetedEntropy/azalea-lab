# Azalea Lab - Minecraft Bot for Modded NeoForge Server

## What This Project Is

A proof-of-concept that connects a **Rust-based Minecraft bot** (using the Azalea library) to a
**heavily modded NeoForge 1.21.1 server** running the DynamicOdyssey modpack.

NeoForge normally rejects any client that isn't also running NeoForge. This project includes
three server-side patches that bypass that validation, allowing the lightweight Rust bot to
connect, join the game, and interact via chat.

### Key Specs

- **Minecraft**: 1.21.1 (protocol 774)
- **Server**: NeoForge 21.1.219
- **Modpack**: DynamicOdyssey (tested with 2.7.0 through 2.26.0)
- **Bot**: Rust + Azalea (patched fork, nightly toolchain)
- **Auth**: Offline mode (no Microsoft/Mojang account needed)
- **Deployment**: Docker

## How to Deploy (Full Steps)

### Prerequisites

| Requirement | Version | Notes |
|-------------|---------|-------|
| Docker | 20.10+ | With Docker Compose v2 |
| Rust | nightly | `rustup toolchain install nightly` |
| RAM | 8 GB+ | Server uses 6 GB; bot needs ~1 GB to compile |
| Disk | ~2 GB | For Docker image build + Rust compilation |

### Step 1: Build the Package from a Modpack Zip

```bash
cd server-deploy

# Pass the server modpack zip вЂ” extracts into pack/ and sets up .env
./build.sh /path/to/DynamicOdyssey-Server-X.Y.Z.zip
```

This extracts the modpack, copies the patches, and auto-detects the NeoForge
version from the installer JAR filename.

### Step 2: Start the Server

```bash
docker compose up -d

# Watch logs until you see "Done (X.XXXs)! For help, type "help""
docker logs -f azalea-mc-server
```

First build takes several minutes (NeoForge installer + mod loading).
Subsequent starts are much faster.

### Step 3: Run the Bot

```bash
# From the server-deploy directory:
./run_bot.sh
```

The bot will:
1. Connect to `localhost:25566` as `azalea_bot` (offline mode)
2. Wait 3 seconds after login
3. Send `"azalea-bot online"` in chat
4. Wait 2 seconds, then exit

### Step 4: Verify

```bash
docker logs azalea-mc-server 2>&1 | grep -E "azalea|joined|chat|online"
```

Expected output:
```
azalea_bot[/...] logged in with entity id ...
azalea_bot joined the game
<azalea_bot> azalea-bot online
azalea_bot left the game
```

## Updating the Modpack

When a new DynamicOdyssey server pack is released:

```bash
cd server-deploy
./build.sh /path/to/DynamicOdyssey-Server-NEW_VERSION.zip
docker compose build
docker compose up -d
```

The build script:
1. Clears the old `pack/` directory
2. Extracts the new zip into `pack/`
3. Auto-detects the NeoForge version from the installer JAR
4. Writes `.env` with `NEOFORGE_VERSION=X.Y.Z` for Docker Compose
5. Copies the three patches from `repo/server/`

**Important**: The patches are built against NeoForge 21.1.219 class signatures.
If the modpack upgrades to a different NeoForge version, the Java patches may need
updating (check compilation errors during `docker compose build`).

## The Azalea Bridge Coremod

A single NeoForge coremod JAR (`azalea-bridge-2.0.0.jar`) dropped into the `mods/`
folder handles all client compatibility via ASM bytecode transformation at class-load
time. **No NeoForge jar modifications or .patch files needed.**

NeoForge 21.1.219 has three layers of validation that reject non-NeoForge clients.
The coremod disables each one:

### Patch 1: NetworkComponentNegotiator.negotiate()

**Problem**: NeoForge requires clients to negotiate mod channels. The bot doesn't
speak the NeoForge channel protocol.

**Fix**: Replaces `negotiate()` body via ASM to always return
`new NegotiationResult(List.of(), true, Map.of())`.

### Patch 2: ConfigurationInitialization.configureModdedClient()

**Problem**: NeoForge runs `RegistryDataMapNegotiation`, `CheckExtensibleEnums`,
and `CheckFeatureFlags` tasks that fail for non-NeoForge clients.

**Fix**: Replaces the entire method body with `RETURN`, skipping all modded
configuration tasks. The bot handles `c:version` and `c:register` in its own
protocol layer.

### Patch 3: NetworkRegistry.checkPacket()

**Problem**: Mods send custom payloads to all clients. `checkPacket()` throws
`UnsupportedOperationException` for unregistered payload types.

**Fix**: Replaces `ATHROW` instructions with `POP + RETURN`, silently ignoring
unknown packets instead of disconnecting.

### Building / Rebuilding the Coremod

**Source**: `repo/server/coremod-jar-v2/`
**Rebuild**: `bash repo/server/build-coremod-v2.sh`

The output `azalea-bridge-2.0.0.jar` goes into `server-deploy/patches/` and is
copied into the Docker image's `mods/` folder during build.

## Bot Architecture

The bot uses **Azalea** (`azalea-rs`), a Rust library for Minecraft clients/bots,
and bridges to **OpenClaw** (self-hosted AI agent gateway) for AI-powered chat.

### Source Layout

```
bot/src/
в”њв”Ђв”Ђ main.rs              # Entry point: config, channel setup, HTTP server + MC bot
в”њв”Ђв”Ђ config.rs            # Environment variable loading
в”њв”Ђв”Ђ state.rs             # BotState (Azalea Component) + SharedState (Arc'd shared data)
в”њв”Ђв”Ђ handler.rs           # Azalea event handler (Login, Chat, Tick, Death, Disconnect)
в”њв”Ђв”Ђ bridge/
в”‚   в”њв”Ђв”Ђ mod.rs
в”‚   в”њв”Ђв”Ђ outbound.rs      # MCв†’OpenClaw: POST /hooks/agent with chat messages
в”‚   в”њв”Ђв”Ђ inbound.rs       # OpenClawв†’MC: axum HTTP server on port 3001
в”‚   в””в”Ђв”Ђ types.rs         # Shared JSON request/response structs
в””в”Ђв”Ђ commands/
    в””в”Ђв”Ђ mod.rs           # BotAction enum + execute() dispatcher
```

### Data Flow

**MC Chat в†’ OpenClaw в†’ MC Chat (someone speaks to the bot):**
1. Player types message mentioning the bot's name in MC chat
2. `Event::Chat` handler detects the mention (case-insensitive)
3. Spawns async task to POST to OpenClaw's `/hooks/agent`
4. OpenClaw processes through LLM, returns reply
5. Bot sends reply as MC chat message

**OpenClaw/Discord в†’ MC (external command to the bot):**
1. OpenClaw POSTs to `http://bot:3001/actions` with `{"action":"chat","message":"..."}`
2. axum handler parses the command, queues a `BotAction` via mpsc channel
3. On next `Event::Tick` (20/sec), handler drains the channel and executes actions
4. Bot sends the message in MC chat

### Bot HTTP API (port 3001)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/health` | GET | Returns `"ok"` if bot is running |
| `/actions` | POST | Execute a bot action (chat, future: teleport, etc.) |

**POST /actions** body:
```json
{"action": "chat", "message": "Hello from Discord!"}
```

Future actions (teleport, look, etc.) will be added as new variants in
`commands::BotAction` and new match arms in `bridge::inbound::parse_action`.

### CLI Flags, Environment Variables, and Config File

Configuration priority: **CLI flags > env vars > config file > defaults**

**CLI flags** (run `./run_bot.sh --help` for full list):
```
-s, --server <HOST>       MC server hostname
-p, --port <PORT>         MC server port
-u, --username <NAME>     Offline mode username
-e, --email <EMAIL>       Microsoft auth email (enables online mode)
-c, --config <PATH>       Path to TOML config file
    --openclaw-url <URL>  OpenClaw gateway URL
    --openclaw-token <T>  OpenClaw bearer token
    --http-port <PORT>    Bot HTTP server port
```

**Environment variables** (same as CLI, picked up automatically):
`MC_HOST`, `MC_PORT`, `BOT_USERNAME`, `MS_EMAIL`, `OPENCLAW_URL`,
`OPENCLAW_TOKEN`, `BOT_HTTP_PORT`, `BOT_CONFIG`, `RUST_LOG`

**Config file** (`azalea-bot.toml`, `config.toml`, or `~/.config/azalea-bot/config.toml`):
```toml
[server]
host = "localhost"
port = 25566

[auth]
mode = "offline"        # "offline" or "microsoft"
username = "azalea_bot" # for offline mode
# email = "user@example.com"  # for microsoft mode

[openclaw]
url = "http://127.0.0.1:18789"
# token = "bearer-token"

[bot]
http_port = 3001
```

See `bot/config.example.toml` for a full template.

### Authentication

**Offline mode** (default): `./run_bot.sh -u azalea_bot`
- No Microsoft account needed, works with `online-mode=false` servers

**Microsoft auth**: `./run_bot.sh -e user@example.com`
- Uses device-code OAuth flow (prints a URL + code to visit)
- Tokens cached at `~/.minecraft/azalea-auth.json` вЂ” subsequent runs skip auth
- Required for `online-mode=true` servers

### Why Patched Forks?

The bot depends on local patched copies of two crates (in `bot/patches/`):
- **azalea** вЂ” Full 17-crate workspace (auth, protocol, physics, inventory, etc.)
  - `azalea-protocol/src/read.rs`: `MAXIMUM_UNCOMPRESSED_LENGTH` bumped from 2MB to
    64MB to handle the large registry sync packets from 300+ mod servers
- **simdnbt** вЂ” NBT parsing library (patched for compatibility)

These are referenced via `path =` dependencies in `Cargo.toml` and `[patch.crates-io]`
overrides to ensure all transitive dependencies also use the patched versions.

### Rust Toolchain

The project requires **Rust nightly** (specified in `bot/patches/azalea/rust-toolchain`).

### Adding New Bot Capabilities

To add a new action (e.g., teleport):

1. Add a variant to `BotAction` in `commands/mod.rs`:
   ```rust
   Teleport { x: f64, y: f64, z: f64 },
   ```
2. Add a match arm in `commands::execute()` with the bot API call
3. Add a match arm in `bridge::inbound::parse_action()` for the JSON mapping
4. Add any needed fields to `InboundCommand` in `bridge/types.rs`

## OpenClaw Integration

The bot integrates with [OpenClaw](https://github.com/openclaw/openclaw), a self-hosted
AI agent gateway that bridges Discord (and other channels) to LLMs.

### Setup

1. Install and run OpenClaw with Discord channel enabled
2. Set `OPENCLAW_URL` and `OPENCLAW_TOKEN` env vars when running the bot
3. The bot POSTs to `/hooks/agent` when mentioned in MC chat
4. OpenClaw can POST to `http://bot:3001/actions` to send commands to MC

### OpenClaw Webhook Payload (bot в†’ OpenClaw)

```json
{
  "sender": "PlayerName",
  "content": "hey azalea_bot what's up?",
  "source": "minecraft",
  "whisper": false
}
```
